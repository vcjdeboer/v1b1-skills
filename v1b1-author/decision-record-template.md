## Step 4 — Backend shape

**Decision**: One Backend per Capability, sharing the driver. Workflow methods on Backend, NOT on Capability frontend. `_on_setup` / `_on_stop` lifecycle hooks for capability-specific init.

**Principle**: P2 — hardware agnosticity at the capability frontend (Capability frontend is hardware-agnostic; vendor specifics live on the Backend); P6 — driver is the wire, backend is the protocol; P-20 — workflow methods sequencing vendor calls belong on the backend; P-25 — lifecycle hook scope.

**v1b1 precedent**: `pylabrobot/hamilton/liquid_handlers/nimbus/pip_backend.py:455-568` — `NimbusPIPBackend.drop_tips` orchestrates `DropTips` vs `DropTipsRoll` decision and full firmware payloads on the backend. The PIP Capability frontend methods are thin `@need_capability_ready` forwards.

**Wrong default avoided**: Workflow sequencing on the Capability frontend (e.g., `Capability.scan_rack` doing `trigger → poll → fetch` orchestration). Documented violation in PR #1009 (Micronic Code Reader) Finding 2 and the Echo branch Finding 3 — both authored by the same contributor, both made the same mistake. The Capability frontend is a thin `@need_capability_ready` forward; workflow lives on the Backend so different backends can pick different completion strategies (polling, callbacks, blocking calls).

**Files this decision shapes**: <one Backend file per Capability identified in step 1 (existing-implementation kind); helper-subsystem files per HelperChoice>

## Step 5 — Device wiring

**Decision**: Constructor builds driver, constructs Backends sharing the driver (P-13), wires `_capabilities`. <Optional Resource integration via multiple inheritance | No Resource integration>.

**Principle**: P-07 — lifecycle hook ordering; P-13 — multi-capability shared driver; P-23 — Resource integration via multiple inheritance.

**v1b1 precedent**: `pylabrobot/qinstruments/bioshake.py:300-326` — `BioShake.__init__` constructs `BioShakeDriver` once, then conditionally wires `Shaker(backend=BioShakeShakerBackend(driver))` and `TemperatureController(backend=BioShakeTemperatureBackend(driver))`. <Optional: + `pylabrobot/tecan/infinite/infinite.py:19-78` for triple-inheritance Resource + Device + HasLoadingTray form when resource_integration=true.>

**Wrong default avoided**: Capability-bypass via Device convenience methods. Documented violation in the Echo branch Finding 5 — `Echo.is_source_plate_present`, `Echo.get_current_source_plate_type` etc. forward to `self.driver.<method>` directly, bypassing the `PlateAccess` capability surface entirely. The right shape: Device convenience methods either (a) cover genuinely device-level concerns that don't fit any Capability (e.g., serial-number query) and delegate to `self.driver`, OR (b) cover Capability-shaped concerns and delegate to `self.<capability>.<method>`. NEVER bypass the Capability when a Capability covers the operation.
